# Round 1 Review — task-08-cron-systemd-tests
**Reviewer:** Codex
**Verdict:** NEEDS_CHANGES

## Findings (≤5)
- HIGH / `kpi/tests/test_install_kpi_vault_script.py:51` / The committed focused suite fails: `test_install_script_references_expected_units` requires literal `kpi-nightly-ingest.service` and `kpi-heartbeat.service` strings, but `install_kpi_vault.sh` only derives those names from `UNITS_TIMER_PAIRS`. This violates the "All tests pass" acceptance criterion and would fail CI even if pytest is available. Suggested fix: either make the test understand the pair expansion or list/copy the five full unit filenames explicitly in the installer.
- HIGH / `kpi/docs/tz/kpi/task-08-cron-systemd-tests.md:9` / The task requires durable CronCreate registrations for `kpi_nightly_ingest` and `kpi_monitoring_heartbeat`, plus `config/scheduled-crons.tson` updates, but commit `60e7771` only changes scripts, systemd units, and tests. The cron half of the dual-trigger acceptance criterion is not implemented or evidenced. Suggested fix: add/update the scheduled-crons entry and include the CronCreate registration evidence/runbook state in the task artifact.
- MEDIUM / `kpi/scripts/install_kpi_vault.sh:74` / The installer enables the service and both timers before validating `/api/health`, then fails if the response is missing or not `{"status":"ok"}`; however `/api/health` returns `down` when there is no `nightly_orchestrator` row yet. A first install before the first successful ingest can therefore abort after leaving timers active. Suggested fix: either probe only endpoint reachability during install and verify `ok` after first ingest, or rollback/disable the units on failed health validation.
- MEDIUM / `kpi/scripts/run_nightly.sh:25` / First-success flags are written before Telegram delivery is known, and `send_telegram_alert` always returns 0 even for malformed `BOT_TOKEN` or curl failure. A bad token on the first success permanently suppresses the required first-success/heartbeat-live alert after the token is fixed. Suggested fix: have the helper return a distinguishable sent/skipped status and write first-run flags only after confirmed delivery, or store a pending-alert state that retries.
- MEDIUM / `kpi/scripts/heartbeat.sh:63` / The heartbeat assumes writable SQLite and does not handle insert failures; a read-only DB, missing `monitoring_pings` table, or schema mismatch produces a traceback in the captured `status` string and the shell still exits 0 without inserting a row or alerting. This conflicts with the heartbeat acceptance criterion and masks the monitoring DB read-only/write-mode mismatch. Suggested fix: catch write/schema errors explicitly, log and alert them as heartbeat failures, and consider returning non-zero when the ping row cannot be persisted.

## Verdict rationale (3-5 sentences)
The systemd unit structure is broadly in line with the requested timer/service split, and shell syntax checks pass, but the task is not shippable as committed. The focused tests added by the commit fail locally, and the cron registration acceptance criterion is missing entirely from the changed files. The installer and alert paths also have first-run failure modes that can leave enabled timers after a failed health check or permanently skip the required Telegram first-success notification. I could not run `systemd-analyze verify --user` in this environment because the user systemd manager is unavailable (`Failed to initialize manager: No such device or address`).
